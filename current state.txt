  1. Open Questions in Full

  These are the 6 open questions from the mapping exercise:

  Q1 — Migration of existing IDs: We currently have BACK-001 through BACK-099, DEC-001 through DEC-0XX, PROJ-001 through PROJ-0XX. When we switch to short UUIDs (e.g.,
   BACK-a7f3b2c1), do we keep existing sequential IDs as-is and only generate UUIDs for new items going forward? Or do we migrate everything at once, replacing all
  sequential IDs with UUIDs? Keeping both means mixed formats in perpetuity. Migrating means rewriting all cross-references across all files.

  Q2 — OWNER.md / TEAM.md: These files currently live at context/OWNER.md and context/work-instructions/TEAM.md. In the new system, roles go to context/items/role/. Do
   we keep OWNER.md and TEAM.md as legacy convenience files (human-friendly entry points)? Make them symlinks to the new location? Or fully retire them and only use
  items/role/?

  Q3 — Event log format: Events are the one primitive where git-native diffs arguably don't matter — they're append-only, never edited, primarily machine-consumed.
  Should we still use JSONL files (git-tracked, monthly sharded) or go SQLite-only for events (gitignored, rebuilt from other sources)? JSONL is portable and diffable.
   SQLite-only means events are ephemeral/derived. This affects whether event history survives a clone.

  Q4 — Artifact self-description: Research reports (e.g., process-ontology-primitives-2026-03.md) and work instructions are both artifacts AND their own content. The
  artifact's metadata and the artifact's body are in the same file. Do we just add artifact-style frontmatter to these files in-place? Or do we create a separate
  pointer file in items/artifact/ that references the real file's path? In-place is simpler but means not all artifacts live under items/artifact/. Separate pointers
  mean two files per research report.

  Q5 — Process definitions vs instances: Template process definitions live in templates/processes/ (e.g., "code review process"). But a running instance — "code review
   for PR #42" — is a specific occurrence. Does each running instance get its own file in items/process/? Or do we track instances as work items with a process:
  PROC-xxx reference in frontmatter? The distinction matters because instances are high-volume (every PR triggers a code review instance) while definitions are
  low-volume.

  Q6 — ID prefix consistency: The mapping uses BACK- for work items (legacy name from "backlog") but the primitive is "Work Item." Should we rename to WORK-?
  Similarly, PROJ- means "project" but the primitive is "Scope/Container" which covers iterations, teams, and areas too. Do we standardize prefixes to match primitive
  names (WORK-, SCOPE-) or keep familiar prefixes (BACK-, PROJ-)?

  2. All 16 Primitives Described

  Primitive 1: Work Item

  The atomic unit of work — something that must be done, tracked, and completed. Every process framework has this: Jira calls it an Issue, SAFe calls it a
  Story/Feature/Epic, manufacturing calls it a Work Order, healthcare calls it a Clinical Order, legal calls it a Case. Work items have a lifecycle (draft through
  done), an owner, a priority, and can form hierarchies (epic > feature > story > task). This is the most fundamental primitive — without it, there is nothing to
  manage.

  Primitive 2: Log Entry / Event

  An immutable record of something that happened at a specific point in time. Events are append-only — you never edit history, you add new events that correct or
  supersede. State changes, comments, measurements, gate evaluations — all are events. The event log is the "source of truth for history." Git commits are events.
  Jira's activity feed is events. A patient's medical chart is events. This primitive enables auditability, metrics computation, and undo/replay.

  Primitive 3: Decision Record

  A record of a choice made — the context, options considered, rationale, and consequences. Decisions explain why things are the way they are. Without them,
  institutional knowledge evaporates. Software uses ADRs (Architecture Decision Records). Legal has rulings. Manufacturing has Engineering Change Notices. aibox
  already has DEC-NNN — this primitive validates that pattern and adds structured fields for alternatives and consequences.

  Primitive 4: Artifact

  Any tangible output produced or consumed by a process. Code, documents, builds, reports, designs, datasets, test results — all artifacts. Artifacts have versions,
  authors, and approval states (draft > review > approved > published). In aibox, every file in context/ is implicitly an artifact. The key insight: artifacts are the
  "nouns" that processes create and deliver. Git already provides versioning; aibox adds typed metadata.

  Primitive 5: Role

  A named set of responsibilities and permissions — a "hat" that a person or agent wears. Roles are NOT people; one person can hold multiple roles, one role can be
  filled by multiple people (or agents). Product Owner, Reviewer, Architect, Operator, Physician — these are roles. aibox needs this for multi-agent scenarios (kaits)
  where agents are assigned roles. The role primitive separates "what needs to be done" from "who does it."

  Primitive 6: Process / Workflow

  A defined sequence of steps, decision points, and rules that transforms inputs into outputs. Processes are the "verbs" — they describe HOW work gets done. Every
  process has a trigger (what starts it), steps, decision points, and an end condition. CI/CD pipelines are processes. Clinical pathways are processes. aibox already
  has process templates (bug-fix, code-review, release). This primitive makes them first-class: composable, versioned, triggerable.

  Primitive 7: State Machine

  A formal definition of allowed states and transitions for an entity's lifecycle. State machines are NOT entities — they're definitions that govern other primitives.
  "Backlog > In Progress > In Review > Done" is a state machine. Kanban board columns are a state machine. Jira's configurable workflows are state machines. The
  universal pattern across all domains: Queue > Active > Validation > Terminal, with escape states (blocked, cancelled). Making these configurable per work-item type
  is what gives Jira its power.

  Primitive 8: Category / Taxonomy

  A classification system for organizing entities into groups. Priority (critical/high/medium/low), type (bug/feature/chore), area (backend/frontend), size (S/M/L/XL)
  — these are all category dimensions. Categories can be flat or hierarchical, exclusive (one priority per item) or non-exclusive (many tags per item). Healthcare has
  ICD-10 codes. Legal has case types. GTD has contexts (@phone, @computer). This primitive replaces ad-hoc labels with user-defined, validated classification.

  Primitive 9: Cross-Reference / Relation

  An explicit, typed connection between two entities. "BACK-001 blocks BACK-002," "DEC-005 supersedes DEC-003," "this story implements that feature." Cross-references
  create the web of relationships that gives context its meaning. Without them, every item exists in isolation. Zettelkasten's core insight: value grows with link
  density. The types are universal: parent-child, blocks, relates-to, duplicates, supersedes, implements, caused-by, references.

  Primitive 10: Checkpoint / Gate

  A designated point where work is evaluated against defined criteria before it can proceed. Code review approval, CI checks passing, QC inspection, surgical safety
  checklist, DMAIC tollgate — all gates. Gates enforce quality, compliance, and alignment. They can be blocking (must pass) or advisory (may pass with waiver),
  automated (CI) or manual (human sign-off). Gate criteria are declarative — a list of conditions that must be true.

  Primitive 11: Metric / Measure

  A quantified observation about a system or process. Lead time, throughput, defect rate, cycle time, WIP count — all metrics. Metrics are derived from events
  (aggregated over time) and provide the feedback signal for improvement. Without metrics, improvement is guesswork. Every domain has time metrics, volume metrics,
  quality metrics, and efficiency metrics. The materializer pattern: raw events > aggregation > metric values.

  Primitive 12: Schedule / Cadence

  A time-based pattern that triggers activities. Two fundamental types: cadences (recurring rhythms — sprints, weekly reviews, daily standups) and deadlines (specific
  dates — release day, filing deadline, contract expiration). Cadences create predictability; deadlines create urgency. Even flow-based approaches like Kanban use
  cadences for retrospectives. Cron expressions are the ultimate schedule primitive.

  Primitive 13: Scope / Container

  A boundary that groups related items and defines a context for work. Projects, iterations, teams, departments, portfolios — all scopes. Scopes nest hierarchically
  (portfolio > project > sprint > task). They answer "where does this belong?" In aibox, the context directory is a scope. PROJECTS.md entries are scopes. For kaits,
  each simulated company project is a scope (repo-per-project). The PARA model maps: Project (time-bounded), Area (ongoing), Resource (reference), Archive (completed).

  Primitive 14: Constraint

  A rule or limit that restricts degrees of freedom. WIP limits, SLAs, specification tolerances, regulatory requirements, budgets, deadlines — all constraints. This
  primitive was NOT in the original hypothesis but emerged strongly from cross-domain analysis. Every domain has hard constraints that shape everything: Kanban's WIP
  limits, manufacturing's specification limits, healthcare's HIPAA, legal's statutes. Constraints bridge process definitions and automated enforcement.

  Primitive 15: Context / Environment

  The ambient information and conditions surrounding work — everything you need to know to be effective. This is the meta-primitive. CLAUDE.md, OWNER.md, PRD.md, work
  instructions, research reports — all context. PMBOK calls it "Enterprise Environmental Factors." GTD calls it "@context." This is arguably aibox's most important
  primitive — structuring context so AI agents and humans can work effectively IS the product's core value proposition.

  Primitive 16: Discussion

  A structured conversation that explores a topic, weighs arguments, and may produce decisions. DISC-001 (the document driving this redesign) is the proof. Discussions
   have participants, related items, research references, and produce decisions. They capture the train of thought — the reasoning process itself, not just the
  outcome. This primitive was added during the DISC-001 work because the act of discussing design choices was itself a process primitive.

  3. Configurable Filesystem Sharding

  Agreed — sharding granularity should be configurable per entity type. Here's my proposal:

  Sharding configuration lives in aibox.toml (or a context config file):

  [sharding]
  # Applies to file-per-entity primitives in context/items/
  # Options: "none", "yearly", "monthly", "weekly", "daily"
  # Default: "none" (flat directory)

  [sharding.work]
  strategy = "monthly"     # context/items/work/2026/03/BACK-xxx.md

  [sharding.events]
  strategy = "monthly"     # context/events/2026-03.jsonl

  Changing strategy is non-destructive. Key design rule: the shard path is computed from created_at in the entity's frontmatter + the current sharding config. When you
   change from monthly to weekly:
  - Existing files stay where they are (no mass rename)
  - New files go to the new shard structure
  - The SQLite index knows the actual path of every file regardless of shard layout
  - aibox reindex or aibox migrate-shards can optionally reorganize existing files to match the new strategy

  This means the filesystem can have mixed sharding — some files under 2026/03/, others under 2026/W13/. That's fine because:
  1. The index is the fast lookup path, not directory browsing
  2. Git doesn't care about directory depth
  3. The glob pattern in aibox sync handles any nesting

  Shard path formats:
  - none: context/items/work/BACK-xxx.md
  - yearly: context/items/work/2026/BACK-xxx.md
  - monthly: context/items/work/2026/03/BACK-xxx.md
  - weekly: context/items/work/2026/W13/BACK-xxx.md
  - daily: context/items/work/2026/03/27/BACK-xxx.md

  5. ID Prefix: Pro/Contra Analysis and Recommendation

  Option A: Typed prefixes (BACK-, DEC-, ROLE-, etc.)

  Pro:
  - Instant recognition. Seeing BACK-a7f3b2c1 in a conversation or log, you immediately know it's a work item. No lookup needed.
  - Collision avoidance. Even if two UUIDs collide across types (astronomically unlikely), the prefix disambiguates. BACK-a7f3b2c1 and DEC-a7f3b2c1 are distinct.
  - Grep-friendly. grep BACK- finds all work item references. grep DEC- finds all decision references. Without prefixes, you'd need the index.
  - Human communication. In standups, discussions, commit messages — "I'm working on BACK-a7f3" is self-describing. "I'm working on a7f3b2c1" requires context.
  - URL/path routing. The CLI can route aibox show BACK-a7f3 without needing to search all entity types.

  Contra:
  - Triple source of truth for type. You identified the core problem: the entity's type is encoded in THREE places: (1) the filename/ID prefix, (2) the type: field in
  YAML frontmatter, (3) the directory path items/work/. If these disagree, which wins? This is a real consistency risk.
  - Rename cost. If we rename the primitive (e.g., "Work Item" to "Task"), every existing ID is wrong. IDs should be opaque and stable.
  - Prefix proliferation. 11 entity types = 11 prefixes. Some are awkward: SCHED-? CON-? GATE-? Users must learn them all.
  - False specificity. The prefix suggests certainty about the type that may not hold. If a "bug" is reclassified as a "feature," the BACK- prefix is still correct
  (both are work items), but if we used BUG- / FEAT- prefixes, the ID would lie.

  Option B: Uniform prefix (e.g., all IDs are just a7f3b2c1)

  Pro:
  - Type lives in exactly one place (frontmatter type: field)
  - No consistency risk
  - IDs are truly opaque and stable forever

  Contra:
  - IDs are meaningless without lookup
  - Grep requires the index
  - Human communication suffers badly

  Option C: Single universal prefix (e.g., CTX-a7f3b2c1 for all)

  Pro:
  - Identifies it as an aibox entity vs random hex
  - Type lives in one place

  Contra:
  - Adds nothing useful over option B
  - Still not self-describing

  My Recommendation: Typed prefixes (Option A) with a clear authority rule

  Use typed prefixes, but establish that the YAML frontmatter type: field is the single source of truth for type. The filename prefix and directory path are derived
  conveniences:

  1. On creation: aibox generates the ID with the correct prefix and places the file in the correct directory.
  2. On inconsistency: If filename says BACK- but frontmatter says type: decision-record, the frontmatter wins. aibox lint detects and flags this. aibox fix renames
  the file.
  3. The ID in frontmatter includes the prefix (id: BACK-a7f3b2c1), so it's redundant with the filename — but this is intentional for recoverability (your point in
  Q6).

  The consistency risk is real but manageable with tooling. The human-readability benefit is too large to give up — especially for AI agents reading context, where
  BACK-a7f3b2c1 in a cross-reference instantly communicates type without needing to resolve the reference.

  6. ID Storage and the Sharding Clarification

  Where do IDs live?

  Both filename and YAML frontmatter. This is intentional redundancy for resilience:

  context/items/work/2026/03/BACK-a7f3b2c1.md
  ---
  id: BACK-a7f3b2c1     # ID inside file
  type: work-item        # type inside file
  title: "..."
  ---

  Recovery scenarios:
  - Filename corrupted/renamed: The id: field in YAML survives. aibox fix restores the correct filename.
  - YAML corrupted: The filename contains the ID and type prefix. aibox fix restores the frontmatter id: field.
  - Both corrupted: Git history recovers the file. The ID exists in cross-references from other entities too.

  This is the same principle as database primary keys being stored in the row even though the index also knows them. The cost (a few bytes of redundancy) is negligible
   compared to the recovery benefit.

  Sharding clarification — you caught a real inconsistency

  You're right. I said "events need monthly sharding" (as JSONL files) AND "only work items need directory sharding" — but events are a different primitive that ALSO
  needs sharding. Let me correct:

  Primitives that need filesystem sharding:

  ┌───────────────────┬───────────────────────────────────────────┬──────────────────┐
  │     Primitive     │                    Why                    │ Default strategy │
  ├───────────────────┼───────────────────────────────────────────┼──────────────────┤
  │ Work Item         │ High volume (potentially thousands)       │ monthly          │
  ├───────────────────┼───────────────────────────────────────────┼──────────────────┤
  │ Log Entry / Event │ Very high volume (thousands/day possible) │ monthly          │
  └───────────────────┴───────────────────────────────────────────┴──────────────────┘

  All other primitives are low-volume (tens to low hundreds) and don't need sharding — a flat directory is fine.

  The confusion arose because events use JSONL (not file-per-entity), so their "sharding" is file-per-time-period rather than directory-per-time-period. But both are
  sharding by time, and both should be configurable per your point in Q3:

  [sharding.work]
  strategy = "monthly"     # file-per-entity in year/month dirs

  [sharding.events]
  strategy = "monthly"     # one JSONL file per month

  7. State Machine YAML: Limitations vs Jira

  What Jira's workflow engine supports

  Jira's workflow system (based on OSWorkflow, later Atlassian's own engine) provides:

  1. States (statuses) — named states in categories (To Do, In Progress, Done)
  2. Transitions — named moves between states, each with:
    - Conditions — who CAN trigger this transition (role-based, group-based, permission-based)
    - Validators — pre-checks that must pass (field required, regex match, custom script)
    - Post-functions — side effects when transition completes (assign to user, set field, send notification, fire webhook, run Groovy script)
  3. Screens — which fields to show/require during a transition (e.g., "Resolution" field appears on transition to Done)
  4. Properties — metadata on states/transitions (e.g., jira.issue.editable = false on Done)
  5. Shared workflows — one workflow used across issue types, with scheme mappings
  6. Global transitions — transitions available from ANY state (e.g., "Cancel" from everywhere)

  What our YAML can express

  transitions:
    - from: in-progress
      to: in-review
      trigger: submit
      guard: "has_field(assignee) && refs_exist(type=blocks, resolved=true)"
      actions:
        - set_field: {reviewed_by: null}
        - emit_event: {type: state-change}
        - notify: {role: reviewer}
      required_fields: [description]  # must be non-empty

  Comparison

  ┌───────────────────────┬────────────────────────────────────────────┬───────────────────────────────────────────────────────────────────────────────────────────┐
  │     Jira Feature      │              YAML Equivalent               │                                        Limitation?                                        │
  ├───────────────────────┼────────────────────────────────────────────┼───────────────────────────────────────────────────────────────────────────────────────────┤
  │ States                │ states: list                               │ None — identical                                                                          │
  ├───────────────────────┼────────────────────────────────────────────┼───────────────────────────────────────────────────────────────────────────────────────────┤
  │ Transitions           │ transitions: list                          │ None — identical                                                                          │
  ├───────────────────────┼────────────────────────────────────────────┼───────────────────────────────────────────────────────────────────────────────────────────┤
  │ Conditions (who can)  │ guard: expression with role checks         │ Need an expression language. Simple string-based DSL works. Less powerful than Jira's     │
  │                       │                                            │ plugin-based conditions.                                                                  │
  ├───────────────────────┼────────────────────────────────────────────┼───────────────────────────────────────────────────────────────────────────────────────────┤
  │ Validators            │ guard: expression + required_fields:       │ Covers 90% of cases. Custom validation scripts would need a plugin/hook mechanism.        │
  │ (pre-checks)          │                                            │                                                                                           │
  ├───────────────────────┼────────────────────────────────────────────┼───────────────────────────────────────────────────────────────────────────────────────────┤
  │ Post-functions        │ actions: list                              │ Can express set_field, emit_event, notify. Cannot run arbitrary code without a            │
  │                       │                                            │ hook/plugin system.                                                                       │
  ├───────────────────────┼────────────────────────────────────────────┼───────────────────────────────────────────────────────────────────────────────────────────┤
  │ Screens               │ required_fields: per transition            │ No UI screens (aibox is CLI), but requiring fields achieves the same data quality goal.   │
  ├───────────────────────┼────────────────────────────────────────────┼───────────────────────────────────────────────────────────────────────────────────────────┤
  │ Global transitions    │ from: [all] or from: "*"                   │ Easy to support in the YAML syntax.                                                       │
  ├───────────────────────┼────────────────────────────────────────────┼───────────────────────────────────────────────────────────────────────────────────────────┤
  │ Groovy scripts        │ Not supported                              │ Real limitation. We'd need a scripting hook (shell command, WASM plugin, or similar).     │
  ├───────────────────────┼────────────────────────────────────────────┼───────────────────────────────────────────────────────────────────────────────────────────┤
  │ Workflow schemes      │ Separate config mapping subtypes to state  │ Need a scheme config, but straightforward YAML.                                           │
  │                       │ machines                                   │                                                                                           │
  └───────────────────────┴────────────────────────────────────────────┴───────────────────────────────────────────────────────────────────────────────────────────┘

  What we can't do (and whether it matters)

  1. Arbitrary scripting in transitions. Jira allows Groovy scripts as post-functions. We'd need a hook mechanism (on_transition: "shell-command") to match this. For
  aibox's use cases, declarative actions (set field, emit event, notify) cover 95% of needs. The remaining 5% could be handled by event-driven hooks — the event log
  fires, and external listeners react.
  2. UI-driven workflow designer. Jira has a visual drag-and-drop workflow editor. aibox is YAML-first. This is a feature, not a limitation — YAML is
  version-controlled, diffable, and AI-readable. But less accessible to non-technical users.
  3. Runtime workflow modification. In Jira, you can edit a workflow and publish it, and in-flight issues adopt the new transitions. With YAML + file-per-entity,
  changing the state machine YAML changes it for new transitions immediately, but you need to handle entities currently in states that the new machine removes. This is
   solvable: aibox lint flags entities in invalid states after a machine change.

  Verdict

  YAML state machines give us ~90% of Jira's flexibility. The gaps are:
  - Arbitrary scripting → solved by event-driven hooks
  - UI designer → not needed for aibox's audience
  - Complex conditions → need a simple expression language (not hard to implement)

  The key Jira feature we MUST have is configurable state machines per work-item subtype (bugs have a different workflow than features). The YAML schema supports this
  naturally:

  # context/schemas/state-machines/bug-workflow.yaml
  name: bug-workflow
  governs: work-item
  applies_to_subtypes: [bug]
  # ...states and transitions...

  8. "Gate evaluations and metric values are events, not state on the entity" — Explained

  This is an architectural choice about where data lives. Let me contrast the two approaches:

  Approach A: State on the entity (what NOT to do)

  # In GATE-a7b8c9d0.md frontmatter
  id: GATE-a7b8c9d0
  name: "PR Merge Readiness"
  last_result: passed              # <-- state on the gate
  last_evaluated: 2026-03-27
  evaluated_for: BACK-a7f3b2c1    # <-- which item?
  history:                          # <-- growing unbounded
    - {date: 2026-03-27, target: BACK-a7f3b2c1, result: passed}
    - {date: 2026-03-26, target: BACK-b2c3d4e5, result: failed}
    - {date: 2026-03-25, target: BACK-c3d4e5f6, result: waived}
    # ... hundreds more ...

  Problems:
  - The gate file grows without bound as evaluations accumulate
  - The gate is evaluated against MANY different work items — which one is "the" result?
  - Every evaluation means editing the gate file → merge conflicts
  - You can't query "show me all gate failures this month" without parsing every gate file

  Approach B: Events (what we DO)

  The gate file is a definition — it describes WHAT the gate checks, not what happened:

  # GATE-a7b8c9d0.md — static definition
  id: GATE-a7b8c9d0
  name: "PR Merge Readiness"
  state: active           # is this gate in use?
  severity: blocking
  criteria:
    - id: tests-pass
      description: "All tests pass"
      automated: true

  Each evaluation is an event in the event log:

  {"id":"EVT-x1","ts":"2026-03-27T14:30:00Z","type":"gate-result","subject":"GATE-a7b8c9d0","data":{"target":"BACK-a7f3b2c1","result":"passed","criteria":{"tests-pass"
  :true,"review-approved":true}}}
  {"id":"EVT-x2","ts":"2026-03-26T10:00:00Z","type":"gate-result","subject":"GATE-a7b8c9d0","data":{"target":"BACK-b2c3d4e5","result":"failed","criteria":{"tests-pass"
  :false}}}

  Benefits:
  - Gate definition stays small and stable (rarely edited)
  - Evaluation history is in the event log (append-only, queryable via SQLite index)
  - You CAN query "all gate failures this month" with one SQL query on the warm index
  - No merge conflicts on gate files
  - Gate evaluations ARE events by nature — they happened at a point in time

  Same logic for metrics

  A metric definition says "lead time = done_at - ready_at, target < 48h." That's stable.

  The metric's VALUE at any point in time is computed from events:
  {"id":"EVT-y1","ts":"2026-03-27","type":"metric-sample","subject":"MET-b8c9d0e1","data":{"value":36,"unit":"hours"}}

  The metric's "current value" and "health status" (green/yellow/red) are computed on read by the materializer from the latest events. They are never stored as a field
   on the metric definition — because:
  - Storing them would mean updating the metric file every time a new sample arrives
  - The "current" value depends on the query time window
  - The warm index can compute this instantly from the events table

  The principle

  Definitions are nouns. Evaluations are verbs. Nouns live in entity files. Verbs live in the event log.

  A gate is a definition (noun). Evaluating a gate is something that happens (verb → event). A metric is a definition (noun). Computing a metric value is something
  that happens (verb → event). This separation keeps entity files small, stable, and focused while the event log captures all the action.
